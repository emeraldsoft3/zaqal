## 🌳 Elaboration & Design Hierarchy

Elaboration with `hierarchy -check -top Core` mapped the physical layout of the processor core, verifying all submodule links and identifying unused blocks.

### 1. Structural Map
The processor was successfully elaborated into two main hierarchical trees under `Core`:
* **`Frontend` (Instruction Fetch & Prediction):** Instantiates branch predictors (`BPU`), queues (`FTQ`), aligners (`IBUF`), instruction caches (`ICache`), and control stages (`IFU` $\rightarrow$ `Predecoder` $\rightarrow$ `RVCExpander`).
* **`Backend` (Out-of-Order Execution):** 
  * Registers: `RegFile` (Integer) & `FPRegFile` (Floating-Point).
  * Rename & Dependency Table: `RenameTableWrapper` (contains Logical-to-Physical translation tables `RenameTable` / `RenameTable_1`) and `BusyTable`.
  * Scheduling: Dual issue schedulers `IssueQueue` & `IssueQueue_1`.
  * Execution Pipeline: `Execute` wraps individual functional execution pipelines: `ALU` (which splits into arithmetic submodules `Adder`, `Logical`, `Shifter`, `Bitmanip`), branch resolution `BRU`, memory operations `LSU`, and arithmetic division (`Divider`, `FPDivider`, `FPMisc`).

### 2. Dead Code Pruning
During hierarchy analysis, Yosys identified and removed unreferenced hardware blocks:
* **Pruned Modules:** `Removing unused module \FPU.` (1 module pruned).
* **Insight:** While the individual floating-point components (`FPDivider`, `FPMisc`, `FPRegFile`) are fully instantiated and utilized within the `Execute` pipeline, the top-level generic `FPU` wrapper module itself was not referenced and was pruned to optimize logic overhead.

## ⚙️ Process Compilation (PROC Pass)

The Yosys `proc` pass processes all behavioral procedural blocks (`always` and `initial` blocks) and converts them into structurally defined logic gates, multiplexers, and flip-flops.

### 1. Compilation Workflow
The process compiler successfully executed the following sub-passes:
* **`proc_clean` & `proc_rmdead`:** Pruned empty switch cases and unreachable dead branches in complex instruction decoding and dispatch trees.
* **`proc_init`:** Extracted hardware initialization values from Chisel's simulation block setups (e.g., initial state registers).
* **`proc_arst`:** Detected and mapped asynchronous reset pathways.
* **`proc_mux`:** Compiled SystemVerilog conditional blocks (`if/else` and `case` statements) into 28,920 generic multiplexers (`$mux` / `$pmux` cells).
* **`proc_dff`:** Extracted sequential logic blocks and mapped them to 7,361 edge-triggered flip-flop registers.

### 2. Synchronous Integrity Check (Latch Audit)
We conducted an audit of the latches inferred by Yosys during the `proc_dlatch` sub-pass. Level-sensitive latches (`$dlatch`) can cause timing hazards and race conditions in high-performance processors:
* **Latches Inferred:** **0**
* **Verification:** All procedural assignments were fully closed, meaning there were no missing `else` branches in combinational `always` blocks.
* **Design Insight:** Chisel-defined storage nodes like `div_rd_latch` and `fpdiv_rd_latch` were correctly inferred and optimized into **actual synchronous registers (edge-triggered D-FFs)** instead of raw level-sensitive latches, preserving design stability.

## 💡 Key Design Insights

1. **Issue Queue & Dependency Checking Dominate Area:**
   The schedulers (`IssueQueue` + `IssueQueue_1`) and `BusyTable` consume nearly **42% of the entire design's logic**. This is expected in a **6-wide out-of-order design** where age comparisons, wakeup tag broadcasts, and dependency matrix tracking scale quadratically with the instruction window size.
   
2. **Multiplexer and CAM Densities:**
   With 28,920 `$mux` cells and 13,625 `$eq` comparators, the bypass networks (routing operands from functional units directly back to reservation stations) and CAM-based tag wakeups dominate the combinational logic area.
   
3. **Register Port Pressure:**
   The `RegFile` contains 5,210 cells to support wide superscalar writeback ports. This logic will be mapped to multi-ported register file SRAMs during physical design.

### 2. Memory Compilation Strategy via OpenRAM
Because compiling register files (`RegFile` & `FPRegFile`) and caches (`ICache` & `DataMem`) out of standard cells (flip-flops and logic gates) leads to extreme routing congestion and compilation timeouts, we use **OpenRAM** to compile dense memory macros.
* **Target macro size for Integer RF:** `64 words` x `64 bits`
* **Outputs Generated by OpenRAM:**
  * **`.lef` (Library Exchange Format):** Contains boundary and metal layer geometries for the macro placement.
  * **`.lib` (Liberty Timing File):** Contains delay model values, setup/hold constraints, and power consumption statistics.
  * **`.gds` (GDSII Data):** Mask layout databases for final tapeout verification.
* **Macro Integration in OpenLane:** Add the compiled macro paths to `EXTRA_LEFS` and `EXTRA_LIBS` in `config.json`, replacing raw Register File behavioral code with Chisel `ExtModule` wrappers.
